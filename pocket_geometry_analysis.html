Do the LMDB pockets agree with the PDBBind combine_set structures?

Consequences for assigning pockets to protein chains and UniProt entries

Published

July 19, 2026

1 Executive answer

Not by PDB ID alone. In these reports, none of the 66,427 LMDB records is an exact full-atom match to the corresponding combine_set/<pdb_id>/data.pkl, and only 220 records (0.33%) are exact matches after hydrogens are removed. The remaining 66,175 records are classified as mismatches, while 32 records refer to eight PDB IDs for which no local reference pocket was found.

This does not imply that every mismatching LMDB pocket is unrelated to the deposited complex. The report uses a strict criterion: atom labels must have the same order and the coordinate arrays must agree element by element. A different subset of atoms from the same protein fails that criterion even when every LMDB coordinate occurs in the deposited protein structure.

The practical conclusion is therefore:

  1. A full_match or heavy_atom_match is sufficient to use the corresponding combine_set structure for chain assignment.
  2. A mismatch must undergo a second, per-LMDB-record spatial containment test against <pdb_id>_protein.pdb.
  3. Only records whose atoms can be located in that protein file should inherit its chain/entity/UniProt annotations.
  4. Records that do not map spatially must not be assigned from the raw PDB ID. Their actual source structure or augmentation provenance is required.
  5. A pocket may span multiple chains. In that case the correct result is a set of candidate proteins/entities, not an arbitrarily selected single protein.

The sections below reproduce the report statistics and implement the required spatial validation.

7 The relevant second test: spatial containment in the protein PDB

For protein assignment, an LMDB pocket does not have to equal the reference pocket. It is enough for its atoms to be found in the deposited protein structure. The function below matches every LMDB coordinate to the nearest ATOM coordinate in <pdb_id>_protein.pdb. It also checks the chemical element and returns the author chain IDs of the matched atoms.

There is an important representation inconsistency in the supplied DrugCLIP data. dict_pkt.txt contains the element tokens C, N, O, S, and H, but the actual pocket_atoms arrays contain PDB atom names such as CA, CB, CG, NZ, and OD1. DrugCLIP’s local PDB writer also reads atom_name into pocket_atoms. The audit therefore reports two separate quantities:

  • direct agreement with the stored PDB atom name, useful for diagnosing the files as they actually exist;
  • chemical-element agreement after normalizing the LMDB label and reading PDB columns 77–78, which is the biologically relevant label check and is consistent with dict_pkt.txt.

The PDB element field is authoritative when present. A fallback infers the element from the aligned four-character atom-name field when columns 77–78 are blank.

PDB coordinates are printed to three decimal places. A 0.002 Å Euclidean tolerance safely covers that rounding while remaining far below ordinary interatomic distances. No structural alignment is performed: both coordinate sets must already occupy the same Cartesian frame.

def find_protein_pdb(pdb_id: str) -> Path | None:
    directory = COMBINE_SET_DIR / pdb_id.lower()
    preferred = directory / f"{pdb_id.lower()}_protein.pdb"
    if preferred.exists():
        return preferred
    alternatives = sorted(directory.glob("*_protein.pdb"))
    return alternatives[0] if alternatives else None


def read_protein_atoms(
    path: Path,
) -> tuple[np.ndarray, np.ndarray, np.ndarray, np.ndarray]:
    coordinates: list[tuple[float, float, float]] = []
    atom_names: list[str] = []
    elements: list[str] = []
    chain_ids: list[str] = []

    with path.open(encoding="utf-8", errors="replace") as handle:
        for line in handle:
            if line[:6].strip() != "ATOM":
                continue
            try:
                xyz = (
                    float(line[30:38]),
                    float(line[38:46]),
                    float(line[46:54]),
                )
            except ValueError:
                continue
            coordinates.append(xyz)
            raw_atom_name = line[12:16]
            atom_name = raw_atom_name.strip()
            atom_names.append(atom_name)
            elements.append(
                (line[76:78].strip() or infer_pdb_element(raw_atom_name)).upper()
            )
            chain_ids.append(line[21].strip() or "?")

    if not coordinates:
        raise ValueError(f"No ATOM coordinates found in {path}")
    return (
        np.asarray(coordinates, dtype=np.float64),
        np.asarray(atom_names),
        np.asarray(elements),
        np.asarray(chain_ids),
    )


TWO_LETTER_ELEMENTS = {
    "BR", "CA", "CL", "CO", "CU", "FE", "MG", "MN", "MO", "NA", "NI",
    "SE", "ZN",
}


def infer_pdb_element(raw_atom_name: str) -> str:
    """Infer an element from the aligned PDB atom-name field."""
    if len(raw_atom_name) != 4:
        raw_atom_name = f"{raw_atom_name:>4}"[-4:]
    stripped = raw_atom_name.strip()
    if not stripped:
        raise ValueError("Cannot infer an element from an empty atom name")

    # A leading space or digit is the usual one-letter-element layout, e.g.
    # " CA " (alpha carbon) or "1HG2" (hydrogen). A left-justified name can
    # represent a two-letter element, e.g. "ZN  ".
    if raw_atom_name[0].isspace() or raw_atom_name[0].isdigit():
        letters = "".join(character for character in stripped if character.isalpha())
        return letters[0].upper()
    candidate = "".join(
        character for character in stripped[:2] if character.isalpha()
    ).upper()
    return candidate if candidate in TWO_LETTER_ELEMENTS else stripped[0].upper()


def element_from_lmdb_label(label: str) -> str:
    """Normalize a stored element token or protein PDB atom name."""
    letters = "".join(character for character in str(label) if character.isalpha())
    if not letters:
        raise ValueError(f"Cannot infer an element from LMDB label {label!r}")
    # LMDB protein labels such as CA, NZ, OD1, and SD are PDB atom names, so
    # their first letter is the element. This also leaves C/N/O/S/H unchanged.
    return letters[0].upper()


def nearest_reference_atoms(
    query: np.ndarray,
    reference: np.ndarray,
    *,
    chunk_size: int = 128,
) -> tuple[np.ndarray, np.ndarray]:
    """Return nearest distances and indices without requiring SciPy."""
    distances = np.empty(len(query), dtype=np.float64)
    indices = np.empty(len(query), dtype=np.int64)
    for start in range(0, len(query), chunk_size):
        stop = min(start + chunk_size, len(query))
        squared = np.sum(
            (query[start:stop, None, :] - reference[None, :, :]) ** 2,
            axis=2,
        )
        local_indices = np.argmin(squared, axis=1)
        indices[start:stop] = local_indices
        distances[start:stop] = np.sqrt(
            squared[np.arange(stop - start), local_indices]
        )
    return distances, indices


def audit_record_against_protein(
    *,
    pdb_id: str,
    split: str,
    lmdb_key: str,
    record: dict,
    tolerance: float = 0.002,
) -> dict:
    protein_path = find_protein_pdb(pdb_id)
    if protein_path is None:
        return {
            "pdb_id": pdb_id,
            "split": split,
            "lmdb_key": lmdb_key,
            "protein_match_status": "protein_pdb_missing",
        }

    (
        reference_xyz,
        reference_atom_names,
        reference_elements,
        reference_chains,
    ) = read_protein_atoms(protein_path)
    query_xyz = np.asarray(record["pocket_coordinates"], dtype=np.float64)
    query_labels = np.asarray([str(x).strip() for x in record["pocket_atoms"]])
    query_elements = np.asarray(
        [element_from_lmdb_label(label) for label in query_labels]
    )
    distances, nearest = nearest_reference_atoms(query_xyz, reference_xyz)
    matched = distances <= tolerance
    matched_count = int(matched.sum())
    atom_names_equal = (
        query_labels[matched] == reference_atom_names[nearest[matched]]
    )
    elements_equal = (
        query_elements[matched] == reference_elements[nearest[matched]]
    )
    chains = sorted(set(reference_chains[nearest[matched]].tolist()))

    if matched_count == len(query_xyz) and bool(elements_equal.all()):
        status = "all_coordinates_and_elements_match"
    elif matched_count == len(query_xyz):
        status = "all_coordinates_match"
    elif matched_count:
        status = "partial_coordinate_match"
    else:
        status = "no_coordinate_match"

    return {
        "pdb_id": pdb_id,
        "split": split,
        "lmdb_key": lmdb_key,
        "protein_match_status": status,
        "lmdb_atom_count": len(query_xyz),
        "matched_atom_count": matched_count,
        "matched_atom_fraction": matched_count / len(query_xyz),
        "matched_atom_name_fraction": (
            float(atom_names_equal.mean()) if matched_count else None
        ),
        "matched_element_fraction": (
            float(elements_equal.mean()) if matched_count else None
        ),
        "maximum_nearest_distance_A": float(distances.max()),
        "matched_auth_asym_ids": chains,
        "protein_path": str(protein_path),
    }

9 Recommended production rule

The safe rule for enriching rankings is:

LMDB record
  ├─ full/heavy exact reference match
  │    └─ use chain(s) from the corresponding combine_set pocket/protein
  └─ strict mismatch
       ├─ all LMDB atoms spatially match <pdb_id>_protein.pdb
       │    └─ use the chain(s) returned by the coordinate matches
       ├─ only some atoms match
       │    └─ flag for review; do not select one protein automatically
       └─ no atoms match
            └─ do not use combine_set for protein assignment

For production use, write the spatial-audit result to a new sidecar keyed by (split, lmdb_key) (or by the candidate-library digest and candidate key). Suggested columns are:

  • protein_match_status;
  • matched_atom_fraction;
  • matched_atom_name_fraction (diagnostic only);
  • matched_element_fraction;
  • maximum_nearest_distance_A;
  • matched_auth_asym_ids;
  • rcsb_polymer_entity_ids;
  • uniprot_accessions;
  • the protein file path and a digest of that file;
  • the coordinate tolerance and algorithm version.

The existing entry-level UniProt metadata remains useful, but it should be filtered through this geometry-to-chain relation before claiming that a specific protein contains a specific LMDB pocket.

10 Final conclusion

The current data support a conditional yes, not an unconditional one:

  • The combine_set structures are suitable for identifying pocket proteins when the individual LMDB geometry can be matched to their atoms.
  • The raw PDB complex ID is not sufficient, because repeated IDs frequently carry distinct geometries and strict reference agreement is very rare.
  • Exact heavy-atom matches are already safe.
  • Many strict mismatches may still be safe after a spatial containment test, because a different pocket subset can belong to the same deposited protein.
  • A mismatch that fails containment must remain unmapped until its true source structure or augmentation provenance is recovered.

Thus, combine_set can be part of the solution, but protein assignment must be made per physical LMDB candidate, not per PDB ID.
